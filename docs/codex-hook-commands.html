[Codex Hook Commands]

scut codex hook <event> handles the command-hook lifecycle documented by Codex. The surface mirrors the Claude hook command pattern where Codex has a published equivalent, while keeping Codex-native event fields such as turn_id, model, and apply_patch tool names.

// 01How It Works

Codex command hooks run as subprocesses. Codex sends one JSON object on stdin, then reads optional JSON from stdout. Hooks can be stored in hooks.json next to an active Codex config layer, or inline under [hooks] in config.toml. Project-local hooks only load once the project .codex/ layer is trusted.

  1. Codex fires an event such as PreToolUse, PostToolUse, or Stop.
  2. The matcher group selects handlers. Tool events match tool_name and aliases such as Edit/Write for apply_patch.
  3. scut codex hook <event> decodes stdin into the event-specific input type.
  4. The handler writes a valid JSON response, usually an empty output object unless the event injects context or blocks/continues.
noteCodex command hooks are trust-gated. Non-managed hooks must be reviewed through Codex's /hooks UI before they run, unless Codex is launched with --dangerously-bypass-hook-trust.

// 02Shared Types

All Codex hook wire types live in the exported hooks/codex package. It is intentionally public for the same reason as hooks/claudecode: other local tools can import the canonical structs instead of rebuilding the JSON shapes.

TypePurpose
Input Common fields: session_id, transcript_path, cwd, hook_event_name, model, and optional permission_mode.
TurnInput Embeds Input and adds Codex's turn_id for turn-scoped events.
BaseOutput Common JSON fields: continue, stopReason, systemMessage, and suppressOutput.
ContextHookOutput Nested hookSpecificOutput shape for event-specific additionalContext.

Polymorphic fields such as tool_input and tool_response use json.RawMessage. The Codex docs describe these as JSON values because Bash, apply_patch, and MCP tools all use different payload shapes.

// 03Command Structure

Commands live in internal/cmd/codex/hook/. Each leaf command has a Run(stdin io.Reader, stdout io.Writer, logger *slog.Logger) error method and embeds trailingArgs so unexpected future positional arguments do not break hook invocation.

The top-level group is scut codex. It has the same --log and --log-level flags as scut claude, and the root command chooses the logger based on the parsed Kong command path.

Codex's current tool hooks intercept Bash, file edits performed through apply_patch, and MCP tool calls. They do not yet intercept every shell or web operation path. The docs call this a guardrail rather than a complete enforcement boundary.

post-tool-use reuses the Claude formatter policy for Codex edit flows. It extracts file_path when a tool supplies one directly, or parses apply_patch text from fields such as command, patch, or input. Relative patch paths are resolved against cwd. Added, updated, and moved files are candidates for formatting; deleted files are skipped.

// 05Claude Parity

The Codex implementation follows the Claude hook architecture where the official Codex surface overlaps: public wire types, a Kong subcommand per event, JSON stdin/stdout, structured logging, and forward-compatible trailing args.

AreaStatus
Implemented The ten documented Codex command-hook events have typed inputs, typed outputs, subcommands, and round-trip tests.
Not implemented Codex config install/uninstall/status commands. Codex uses hooks.json and inline [hooks], which should be modelled separately from Claude's settings.json.
Not a Codex feature today Claude-only events such as Notification, SessionEnd, PostToolUseFailure, worktree hooks, MCP elicitation hooks, and team/task events are not added to scut codex until Codex documents them.

// 06Current Status

All ten Codex hook commands currently deserialize their stdin payloads and emit valid JSON. post-tool-use has a real formatter implementation: it formats .go, .md, and .mdx files, respects root .prettierignore and .scutignore, and returns nested additionalContext only when it changes files. session-start, subagent-start, and user-prompt-submit return placeholder additionalContext so their nested output shape is exercised. Remaining tool and stop hooks return empty outputs until repository-specific policy is added.
