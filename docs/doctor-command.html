[scut doctor]

scut doctor diagnoses local hook setup without writing files. It checks global command availability, Claude Code settings, Codex hook files, Codex TOML conflicts, and generated hook command shapes.

// 01Overview

The command is intentionally read-only. It does not call scut init, does not repair files, and does not create missing directories. Its job is to explain whether the current setup looks usable and where the user should look next.

doctor.sh SH 
# Check Claude Code and Codex in project + user scopes
scut doctor

# Check only Codex project setup
scut doctor --codex --scope=project

# Emit machine-readable diagnostics
scut doctor --json

// 02Checks

CheckBehavior
scut-path Verifies scut is discoverable on PATH, because generated hook commands use bare scut.
Claude settings Looks for .claude/settings.json and ~/.claude/settings.json, parses JSON when present, and reports scut-owned hook/status entries.
Codex hooks Looks for .codex/hooks.json and ~/.codex/hooks.json, parses JSON when present, and reports scut-owned hook entries.
Codex config.toml Warns when inline [hooks] exists beside hooks.json and errors when hooks = false or codex_hooks = false appears.
project-trust When project .codex/ exists, emits an informational reminder that Codex project hooks require the project layer to be trusted.

// 03Output

Human output uses four severities: OK, INFO, WARN, and ERROR. JSON output returns the same findings as an array of objects with severity, agent, scope, check, path, and message. The command exits non-zero when any finding has error severity.
