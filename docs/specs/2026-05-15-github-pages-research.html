// 02Source Options

GitHub Pages currently supports exactly two publishing source modes. They are mutually exclusive — you pick one in Settings and it governs how all deployments work. (source: docs.github.com)

ModeHow it worksOur fit
Deploy from a branch On each push to the chosen branch, GitHub runs a built-in pipeline that passes the selected folder (/ root or /docs) through Jekyll and publishes the result. No workflow file needed. Works, but we get Jekyll processing we don't need, 10-build/hour rate limit applies, and we cannot control the trigger.
GitHub Actions A workflow file in .github/workflows/ drives everything: checkout, optionally transform files, upload an artifact via actions/upload-pages-artifact, deploy via actions/deploy-pages. The branch-rate limit does not apply. More explicit, allows paths: filters, no Jekyll unless we invoke it, artifact approach cleanly separates what we publish from the rest of the repo.

GitHub's stated guidance

The official docs phrase it as: "If you do not need any control over the build process for your site, we recommend that you publish your site when changes are pushed to a specific branch. If you want to use a build process other than Jekyll or you do not want a dedicated branch to hold your compiled static files, we recommend that you write a GitHub Actions workflow." (source)

For our case — no build step, no Markdown, HTML files that reference assets by relative path — either mode technically works. The key differentiators are:

  • Branch mode runs Jekyll by default. We can suppress it with .nojekyll but that's an extra file to maintain. The Actions mode with no Jekyll invocation simply never runs it.
  • Branch mode allows only / root or /docs as the source folder. Both match our repo layout (we'd use /docs). Actions mode uses the path: parameter on upload-pages-artifact for the same effect.
  • Branch mode triggers on every push that touches any file in the branch, not just docs/. Actions mode allows paths: ['docs/**'].
noteThe official starter workflow for static sites (retrieved 2026-05-15) shows a single-job pattern with no separate build/deploy split: checkout → configure-pages → upload artifact → deploy. For a site with no build step this is fine.

// 03Project Page URLs

Because ajbeck/scut is a project repository (not a ajbeck.github.io repository), Pages publishes under a subpath. (source: docs.github.com — types of sites)

Site typePublished URL
User page (repo: ajbeck.github.io) https://ajbeck.github.io/
Project page (repo: ajbeck/scut) https://ajbeck.github.io/scut/

The subpath /scut/ is a URL prefix, not a filesystem directory. Whether paths resolve correctly under it depends entirely on how HTML files reference their assets. There are two categories:

Reference styleExampleResolves under /scut/?
Absolute (slash-prefixed) /scut-docs.css Breaks. Browser resolves to https://ajbeck.github.io/scut-docs.css — wrong host root, 404.
Relative (no slash) scut-docs.css Works. Resolves relative to the current document's URL. If the document is at /scut/kong-base-setup.html, the browser requests /scut/scut-docs.css.
Parent-relative ../scut-docs.css Works. From /scut/specs/2026-05-14-foo.html, resolves to /scut/scut-docs.css.

Our HTML files already use relative paths (scut-docs.css from top-level, ../scut-docs.css from specs/). Neither uses a leading slash. This means our asset references are correct for a project page without any modification.

noteThe same analysis applies to inter-document links (e.g. <a href="logging.html"> or <a href="../kong-base-setup.html">). As long as we do not use absolute paths, the subpath does not break anything. Verify no href="/" or src="/ appears in our HTML before deploying.

// 04Jekyll Behavior

By default, when using branch-mode publishing, GitHub Pages passes the source folder through Jekyll before serving it. For an HTML-only site this has two relevant effects:

Default Jekyll behaviorImpact on our site
Jekyll processes HTML files through its template engine Our HTML contains no Jekyll front matter (--- blocks), so Jekyll passes HTML files through unmodified. No template expansion will fire.
Jekyll ignores _-prefixed files and directories Our docs/ has no _-prefixed files. Not a current problem, but any future _drafts/ or _data/ directory would be silently excluded from the published site. (source)
Jekyll ignores .-prefixed files Dotfiles in docs/ (e.g. .nojekyll itself) are excluded by Jekyll but the file still works as a signal to the pipeline.

The .nojekyll file

An empty file named .nojekyll placed at the root of the publishing source tells GitHub to skip Jekyll entirely. (source: docs.github.com) The source files are served verbatim.

noteFor our current layout — no _-prefixed files, pure static HTML — .nojekyll is not strictly required. Jekyll will pass our files through without modification. Adding it is defensive: it prevents Jekyll processing overhead, avoids any future surprise if we add a file starting with _, and makes intent explicit. The cost is one empty file in docs/.
noteIf we use the GitHub Actions approach (not branch mode), Jekyll never runs regardless. .nojekyll only matters for branch-mode publishing. The Actions artifact upload bypasses Jekyll entirely.

Trade-off summary

ScenarioNeed .nojekyll?Risk without it
Branch mode, no _ files today Optional Low. Future _-prefixed file would silently disappear from the published site.
Branch mode, with _ files Required Files starting with _ are excluded from the published output.
Actions mode Irrelevant Jekyll does not run. The artifact contents are deployed as-is.

// 05Actions Workflow Components

The official Actions-based approach uses three purpose-built actions provided by the actions/ organization. All three are required for a complete deploy. Versions retrieved 2026-05-15.

ActionLatest versionRoleRequired?
actions/configure-pages v6.0.0 (2026-03-25) Enables Pages on the repo if not already enabled (enablement: false by default). Extracts metadata: base_url, origin, host, base_path. Also configures supported SSGs (Next.js, Gatsby, Nuxt, SvelteKit). For a plain static site, this action's primary value is the base_path output — useful if you want to inject the subpath into build output. Not strictly required for a site that already uses correct relative paths. Optional for pure static
actions/upload-pages-artifact v5.0.0 (2026-04-10) Packages a directory into a github-pages artifact and uploads it. The path: input (required; defaults to _site/) specifies which directory to package. The contents of that directory become the root of the deployed site — the directory name itself is stripped. Artifacts expire after 1 day by default. Required
actions/deploy-pages v5.0.0 (2026-03-25) Deploys the artifact produced by upload-pages-artifact to the Pages CDN. Outputs page_url (the deployed URL). Must run in the deploy job, which depends on the build job via needs:. Required

Permissions and environment

The deploy job requires specific GITHUB_TOKEN permissions and must declare the github-pages environment. (source: actions/deploy-pages README)

permissions-block.yml YAML 
# Top-level permissions (or per-job if you use a split build/deploy structure)
permissions:
  contents: read
  pages: write
  id-token: write  # OIDC token; required by deploy-pages for verification

The deploy job must also declare the deployment environment:

deploy-job-environment.yml YAML 
jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

The github-pages environment is created automatically the first time a workflow runs, if it does not already exist. GitHub recommends adding a deployment protection rule so only the default branch can deploy to it.

Full single-job starter workflow (official, static sites)

This is the exact content of actions/starter-workflows/pages/static.yml as retrieved 2026-05-15. Note the action versions used here (configure-pages@v5, upload-pages-artifact@v3, deploy-pages@v5) are older than the latest available; the next session should pin to current versions.

.github/workflows/static.yml YAML 
name: Deploy static content to Pages

on:
  push:
    branches: [$default-branch]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Setup Pages
        uses: actions/configure-pages@v5
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          # Upload entire repository
          path: '.'
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v5
warnThe starter workflow above uses path: '.' — the entire repository root. For our use case we want path: './docs' so only the docs/ directory is published. Using path: '.' would publish Go source code, Magefiles, and everything else alongside the HTML docs. This is the first change we must make.

Two-job vs single-job pattern

For sites that have a build step, the canonical pattern splits into a build job (which runs the build and uploads the artifact) and a separate deploy job (needs: build). The deploy job only runs the actions/deploy-pages step. For our site — no build step — a single-job pattern is simpler and the official starter workflow demonstrates it. Either pattern is valid.

// 06Repo Settings Prerequisites

Before any workflow can successfully deploy to Pages, a one-time manual step in the repository settings is required.

  1. Go to Settings → Pages (under "Code and automation").
  2. Under Build and deployment → Source, select GitHub Actions (not "Deploy from a branch").
  3. Save. GitHub creates the github-pages environment automatically the first time a workflow runs.
warnIf you push the workflow file before setting Source to "GitHub Actions" in Settings, the workflow will run but the deploy step will fail because the github-pages environment does not yet exist. The setting must be changed first, or at minimum before the first deploy run. (source)

There are no other required one-time steps for a public repository. The pages: write permission in the workflow grants the GITHUB_TOKEN the right to deploy. No additional secrets or tokens are needed.

Optionally, after the first deploy, navigate to Settings → Environments → github-pages and add a deployment protection rule restricting deployments to the main branch. This prevents accidental deploys from feature branches if the workflow's branches: filter ever changes.

// 07Path / URL Gotchas for Our Layout

This section addresses our exact directory structure and how each piece resolves under the Actions deploy approach (path: './docs').

How upload-pages-artifact serves a subdirectory

When you pass path: './docs', the action packages the contents of docs/ — not the directory wrapper itself — into the artifact. After deployment, the URL root of the Pages site maps directly to the contents of docs/. (source: GitHub Marketplace; secondary confirmation)

Concretely:

File in repoPublished URL
docs/kong-base-setup.html https://ajbeck.github.io/scut/kong-base-setup.html
docs/scut-docs.css https://ajbeck.github.io/scut/scut-docs.css
docs/specs/2026-05-14-config-command.html https://ajbeck.github.io/scut/specs/2026-05-14-config-command.html
docs/scut-docs.js https://ajbeck.github.io/scut/scut-docs.js

Asset path resolution analysis

HTML fileAsset referenceResolves toCorrect?
kong-base-setup.html (top-level) scut-docs.css https://ajbeck.github.io/scut/scut-docs.css Yes
kong-base-setup.html (top-level) scut-docs.js https://ajbeck.github.io/scut/scut-docs.js Yes
specs/2026-05-14-config-command.html ../scut-docs.css https://ajbeck.github.io/scut/scut-docs.css Yes
specs/2026-05-14-config-command.html ../scut-docs.js https://ajbeck.github.io/scut/scut-docs.js Yes

Our existing relative-path convention is correct for both the Actions deploy and the branch-mode /docs deploy. No HTML files need to change.

The one absolute-path risk

Any link or asset reference that starts with / (e.g., /scut-docs.css, /kong-base-setup.html) would resolve to https://ajbeck.github.io/scut-docs.css — the user's GitHub Pages root, not the project page root. This would 404. The fix before deploying is to scan all HTML for href="/", src="/", and similar patterns and ensure none exist.

noteThe actions/configure-pages action outputs a base_path value (/scut for our repo). This is useful if you want to inject the correct base path into HTML at build time (e.g., via a <base href> tag). For a pure static site where all paths are already relative, you don't need it — but it's available if we later add an absolute path anywhere.

Does docs/ need an index.html?

GitHub Pages will serve a directory listing if there is no index.html at the root. Currently our docs/ has no index.html. Navigating to https://ajbeck.github.io/scut/ would return a 404 (Pages does not generate directory listings). This is a gap we need to address — either add an index.html landing page to docs/, or accept that the root URL produces a 404 and users bookmark specific pages directly. Neither approach is technically broken; it's a UX decision.

warndocs/ currently has no index.html. Without one, the root URL https://ajbeck.github.io/scut/ returns a 404. This must be resolved before the site is usable. Adding a minimal landing page or redirect is the standard fix.

// 08Workflow Triggers

The idiomatic trigger for a Pages deploy workflow is:

trigger-options.yml YAML 
on:
  push:
    branches: [main]
    paths:
      - 'docs/**'
  workflow_dispatch:  # allows manual trigger from Actions tab
Trigger optionBehaviorRecommendation
Every push to main (no paths filter) Deploys on any commit — Go code changes, Magefiles, README updates. Wastes Actions minutes, clutters deployment history. Avoid
paths: ['docs/**'] Deploys only when something under docs/ changes. Commits touching only Go source do not trigger a deploy. Idiomatic; recommended
workflow_dispatch Allows manual one-click deploy from the Actions tab. Useful for first deploy, hotfixes, or debugging. Always include alongside the push trigger

Combining paths: and branches: is AND logic: the workflow runs only when the pushed branch is main AND at least one changed file matches docs/**. (source: docs.github.com — triggering a workflow)

noteIf a push to main changes both Go source and a docs file, the workflow fires (because at least one path matches). This is correct behavior for our case — we want to deploy whenever docs change.

// 10Custom Domains

Covered briefly for completeness. Not in scope for the initial deploy. (source: docs.github.com)

ScenarioDNS recordGitHub-side config
Subdomain (e.g., docs.scut.dev) CNAME → ajbeck.github.io Set custom domain in Settings → Pages; GitHub creates/manages a CNAME file in the repo root (branch mode) or you include it in the artifact (Actions mode).
Apex domain (e.g., scut.dev) A records → four GitHub IPs Same setting; GitHub IPs documented at time of setup (can change — CNAME to subdomain is more stable).

HTTPS enforcement is toggled in Settings → Pages alongside the custom domain field. It can take up to 24 hours to become available after DNS propagation. GitHub auto-provisions a TLS cert via Let's Encrypt.

If using Actions mode and a custom domain: include a CNAME file containing the bare domain name in docs/ so it is included in the uploaded artifact. GitHub does not create it automatically for Actions-based deploys.

noteA custom domain changes the URL root from ajbeck.github.io/scut/ to, say, docs.scut.dev/. This eliminates the subpath prefix entirely. Relative paths in our HTML (scut-docs.css, ../scut-docs.css) work correctly under both URL structures — no HTML changes needed when switching to a custom domain.

// 11Recent Changes Worth Knowing

Artifact actions v3/v4 requirement (January 2025)

On January 30, 2025, GitHub deprecated the use of older artifact action versions in Pages workflows. Any workflow using actions/upload-pages-artifact below v3 or actions/deploy-pages below v4 stopped working on GitHub.com. (source: GitHub Changelog, 2024-12-05)

This is relevant to us only because community tutorials and blog posts written before late 2024 may show older version pins. When writing the workflow, use current versions:

ActionMinimum requiredLatest as of 2026-05-15
actions/upload-pages-artifact v3 v5.0.0
actions/deploy-pages v4 v5.0.0
actions/configure-pages v5 (starter workflow) v6.0.0
warnThe official starter workflow at actions/starter-workflows/pages/static.yml (as retrieved 2026-05-15) still pins upload-pages-artifact@v3 and deploy-pages@v5, which are not the latest versions. The next session should verify the versions to use and pin to current major versions, not blindly copy the starter workflow.

No other significant recent changes found

A search of the GitHub Changelog for 2025–2026 found no Pages-specific behavioral changes beyond the artifact deprecation above. The branch-mode vs Actions-mode choice, URL structure for project pages, and Jekyll/nojekyll behavior have been stable. The Actions pricing changes (January 2026) reduced the cost of hosted runners but do not affect Pages workflow mechanics.
