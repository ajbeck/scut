[release workflows]

scut uses three GitHub Actions workflows: one pull request entry point, one reusable build workflow, and one release workflow that creates tags, publishes assets, updates the Homebrew tap, and deploys the documentation site.

// 01Overview

The workflows are deliberately thin around Mage. GitHub Actions owns orchestration, permissions, artifacts, and release publishing. Mage owns every Go build, format, vet, and test command so the repository's GOEXPERIMENT=jsonv2 setup stays consistent locally and in CI.

WorkflowResponsibility
pull-request.yaml Runs on PRs to main and calls the reusable build workflow in PR mode.
build.yaml Reusable workflow for verification, platform builds, tarball assembly, and checksum generation.
release.yaml Manual release workflow. Tags main, builds from the exact tag, creates the GitHub Release, dispatches the Homebrew tap update, and deploys Pages.

// 02Pull Request

The pull request workflow triggers for PRs targeting main. It has read-only permissions and a concurrency group based on the workflow name and PR ref, with cancel-in-progress enabled. Pushing a new commit to a PR cancels the older PR workflow run and starts a fresh one.

pull-request.yaml 
jobs:
  build:
    uses: ./.github/workflows/build.yaml
    with:
      mode: pr
      ref: ${{ github.sha }}

// 03Build

The build workflow supports both workflow_call and manual dispatch. In PR mode it runs verification first, then builds the four release platforms. In release mode verification is skipped because the release commit has already passed PR checks on main.

  1. verify checks formatting, vet, and tests through Mage. Every Go setup step enables the built-in Go cache and keys dependency invalidation from go.sum.
  2. build runs a matrix for darwin/amd64, darwin/arm64, linux/amd64, and linux/arm64. Each matrix job calls mage buildPlatform <goos> <goarch>.
  3. assemble downloads the platform artifacts, writes checksums.txt, and uploads a single release-assets artifact for release publishing.

The platform artifacts are named scut-<version>-<goos>-<goarch>.tar.gz. Each tarball contains one executable named scut.

// 04Release

The release workflow is manually dispatched with releaseVersion and targetSha. It always checks out main, and targetSha must be the current origin/main commit; releases are not cut from feature branches or an implicit moving HEAD.

  1. tag-release validates releaseVersion with the SemVer.org suggested regular expression, rejects prerelease/build metadata for stable releases, validates full 40-character targetSha, verifies that the target commit matches origin/main, creates immutable vM.m.p when it does not already exist, and force-updates movable vM.m and vM tags to that explicit commit.
  2. build calls the reusable build workflow in release mode with ref set to the exact tag and release_tag set to vM.m.p.
  3. release downloads release-assets, creates the GitHub Release for the exact tag, and uploads tarballs plus checksums.txt.
  4. update-homebrew-tap mints an installation token for the release publisher GitHub App and invokes ajbeck/homebrew-tap's update-formula.yml workflow through workflow_dispatch with formula=scut and version=vM.m.p.
  5. deploy-docs deploys docs/ to GitHub Pages from the same release commit, keeping hosted documentation aligned with the release.
guardrailIf vM.m.p already exists at a different commit, the workflow fails. If it exists at the same commit, the workflow can continue, which allows rerunning a failed release after tagging succeeded.
secretsThe tap dispatch uses AJBECK_RELEASE_PUBLISHER_CLIENT_ID and AJBECK_RELEASE_PUBLISHER_KEY. The GitHub App client secret is not needed for installation-token auth. The GitHub App installation must have Actions: write permission on ajbeck/homebrew-tap so it can call the workflow dispatch endpoint.

// 05Versioning

Release versions come from tags. internal/version.Version defaults to v0.0.0-dev, and release builds override it through linker flags by setting RELEASE_VERSION=vM.m.p. BUILD_METADATA=gh:<run>:<sha> is injected separately.

Source installs such as go install github.com/ajbeck/scut@vM.m.p use Go build information as a fallback when linker flags are not set. Local Mage builds report the development version plus local timestamp metadata.

// 06Dependabot

.github/dependabot.yml checks Go modules and GitHub Actions weekly. Updates are grouped per ecosystem so routine dependency churn lands as focused PRs instead of one PR per dependency.
