[Gotools Doc Plan]

scut gotools doc is a Go documentation command for agents. It keeps the good parts of go doc — parsing and rendering with public Go packages — while adding source resolution for packages that are not required by the current project. The implementation must not shell out to go, git, or go doc.

// 01Goal

Add scut gotools doc, a better Go documentation command that can read documentation for packages outside the current module without mutating the project go.mod and without invoking the Go toolchain as a subprocess.

target-cli.txt ASCII 
scut gotools doc encoding/json Marshal
scut gotools doc github.com/ajbeck/goldmark-adf
scut gotools doc github.com/ajbeck/goldmark-adf Parser --module-version=v0.1.0
scut gotools doc github.com/private/module/pkg Client
scut gotools doc ./internal/godoc Client --src

The first release should be useful for agent workflows, including private modules. It does not need every cmd/go resolver edge case, but it must support authenticated private Git repositories without shelling out. The resolver boundary keeps fuller Go parity additive rather than forcing a rewrite.

// 02Architecture

The implementation splits into four units: command wiring, source resolution, package parsing, and text formatting. The source resolver returns source bytes; the parser turns those bytes into *doc.Package; the formatter emits terminal text.

LayerResponsibilityPrimary APIs
CLI Expose scut gotools doc through Kong and pass stdout/options into the engine. internal/cmd/gotools
Resolver Route package lookups across local module, standard library, module cache, and public module proxy. afero.Fs, net/http, x/mod/module
Parser Parse Go source bytes and build documentation data without OS-path assumptions. go/parser, go/doc, go/token
Formatter Render package, symbol, --all, --short, and --src views. go/format, go/doc/comment
decisionDo not import cmd/go/internal/doc, cmd/go/internal/modfetch, or cmd/go/internal/vcs. They are internal to the Go source tree. We use public Go packages and implement the resolver pieces we need.

// 03Scope

AreaIn v1Deferred
Lookup syntax Full import paths, local ./ paths, optional second symbol argument. Full go doc shorthand search across external modules.
Sources Local module, stdlib, existing module cache, public module proxy, private Git repositories. Workspaces, vendor mode, non-Git VCS backends.
Filesystem All package source reads go through afero.Fs; proxy zips and private Git checkouts extract into afero.NewMemMapFs(). Go checksum metadata writes, cache eviction, and full module-download cache parity.
Resolver policy Respect escaped module paths and versions, explicit version queries, GOPRIVATE routing, shared ?go-get=1 discovery, HTTPS Git auth, longest-prefix package-to-module probing, and best-effort cache write-back through afero.Fs. GOPROXY comma/pipe semantics, GONOPROXY, GOSUMDB, complete GOVCS.
prototype migration checklistUse the robo-duck implementation as the prototype baseline. Before implementation is considered complete, scut must retain the useful prototype behavior for ?go-get=1 discovery, module-cache probing, cache write-back, private Git routing, parser/formatter coverage, and routing tests. The intentional changes are: replace roboduck-specific types with scut types, move filesystem access behind afero.Fs, and remove installed ~/.roboduck fallback behavior.

// 10Task 6 — Proxy Sources

Implement public module proxy resolution. This is the first external-source path and must stay isolated enough that private Git support can be added as another fetcher without changing parser or formatter code.

  1. Create ProxyFetcher with injectable *http.Client and ProxyURL.
  2. Create shared discovery.go support for ?go-get=1 meta tag discovery. The proxy fetcher should try meta discovery before prefix probing so custom module roots resolve the same way the robo-duck prototype does.
  3. Resolve latest through /@latest; resolve explicit versions through /@v/<version>.info.
  4. Probe module prefixes from longest to shortest until a module resolves.
  5. Download .zip, extract matching package files into afero.NewMemMapFs(), then read back through readGoFiles.
  6. Call WriteCache after a successful proxy fetch when a cache directory is configured. This is best-effort: cache write failure must not fail documentation lookup.
  7. Test with httptest.Server and real module-zip entry layout, including meta discovery, prefix probing fallback, cache write-back, and no-cache behavior.
  8. Run mage test.
  9. Commit with feat: resolve Go docs from module proxy.
constraintDo not implement GOPROXY list fallback or sumdb verification in this task. The task is successful when a single configured proxy can resolve and load public module source into memory.

// 11Task 7 — Private Git

Private Git support is part of v1 because some target packages live in private repositories. Keep this as a source fetcher, not as command logic. The fetcher may use existing go-git dependency support, but it must not invoke the git binary.

Concernv1 choice
Routing Use GOPRIVATE glob matching via module.MatchPrefixPatterns. Private matches route to Git before public proxy.
Discovery Try authenticated ?go-get=1 meta tag discovery first; fall back to host convention for common Git hosts using the first three path components.
Authentication Read credentials from environment variables first: GITHUB_TOKEN, then GIT_TOKEN. Use HTTPS basic auth with username token.
Storage Clone with go-git into memory, then copy the target package directory into afero.NewMemMapFs() so parser input still uses the same filesystem boundary.
Version If --module-version is not latest, try it as a tag reference first. latest means default branch HEAD.
  1. Create GitFetcher with injectable auth provider and repository cloner interfaces so tests do not require network access.
  2. Implement isPrivate(pkg string) using GOPRIVATE and module.MatchPrefixPatterns.
  3. Reuse the shared discovery.go ?go-get=1 parser and host-convention fallback, but remove roboduck secret-store coupling.
  4. Clone private Git repositories with go-git in memory, then write matching package files into afero.NewMemMapFs().
  5. Call WriteCache after a successful private Git fetch only when the lookup used a concrete version tag. Do not cache default-branch HEAD as latest.
  6. Test auth selection, private routing, meta discovery, fallback repo split, version tag selection, concrete-version cache write-back, and no-Go-file errors with fake cloner implementations.
  7. Run mage test.
  8. Commit with feat: resolve Go docs from private git.
securityDo not log tokens, repository URLs with embedded credentials, or HTTP authorization headers. Error messages should name the package path and high-level failure, not secret values.

// 15Future Work

These items are intentionally not part of the first implementation. They are resolver extensions beyond the private Git support required for v1.

FeatureWhy deferredLikely implementation
Full GOPROXY semantics Comma vs pipe fallback and direct/off policy should be implemented carefully. Introduce a ProxyList parser and retry loop.
Non-Git VCS The v1 private path targets Git repositories. Mercurial, SVN, and Bazaar parity would increase scope without helping the immediate need. Add additional fetchers behind the same SourceFetcher interface if a real package requires them.
Complete GOVCS policy v1 can avoid shelling out and restrict private fetches to Git. Full Go parity requires parsing and enforcing the entire policy grammar. Implement a GOVCS parser once multiple VCS backends exist.
Checksum database Useful for security parity, but separate from private Git support and not needed to parse docs from a trusted configured proxy in v1. Use x/mod/sumdb APIs after proxy fetches.
Workspace and replace support Needs go.work and replace interpretation across local paths. Extend local resolver with x/mod/modfile.ParseWork and Replace rules.

Implementation plan for scut gotools doc. Keep source resolution behind afero.Fs; keep command wiring thin.
