[Claude Config Command]

scut claude config manages Claude Code's settings.json on behalf of the user. install writes (or merges) every scut hook handler and the status line into the chosen scope. uninstall reverses the operation. status reports what is currently wired. The command never destroys foreign configuration — keys it does not own are round-tripped verbatim.

3subcommands
25hook events wired
2scopes (project, user)
1file touched per run

// 01Purpose

Today, wiring scut into Claude Code is a manual JSON edit. The README documents ~150 lines of settings.json snippets that the user must hand-paste into .claude/settings.json (project scope) or ~/.claude/settings.json (user scope). This is error-prone, hard to keep in sync with new hook events, and impossible to undo without another hand-edit.

scut claude config replaces those hand-edits with three explicit operations. It owns the slot in claude.Cmd already advertised in cmd/scut/main.go:21 ("Claude Code agent commands — hooks, status line, and configuration").

noteThe hook subcommand tree already exists. This command only generates the JSON wiring that invokes those subcommands. It does not modify or replace any hook handler.

// 02Command Tree

command-tree.txt ASCII 
scut claude config
   ├── install     Write/merge scut entries into settings.json
   ├── uninstall   Remove scut entries from settings.json

The new group is a sibling of hook and status-line under claude.Cmd. It mirrors the existing pattern documented in kong-base-setup section 04: a parent Cmd struct with cmd:"" tagged children, each child a leaf with a Run() method.

// 04install

scut claude config install writes scut's hook and status-line entries into settings.json, merging with any existing content.

Flags

FlagTypeDefaultEffect
--scope enum {project, user} project Target file: .claude/settings.json (project) or ~/.claude/settings.json (user)
--only []string (comma-sep) empty = all Whitelist of items to install. Tokens: any hook event slug (post-tool-use, session-start, …) and/or status-line
--log bool false Bake --log into generated command strings (enables JSONL logging at info level)
--log-level string empty Bake --log-level=LEVEL (implies --log). One of debug, info, warn, error
--dry-run bool false Write the resulting JSON to stdout and do not touch the file. Exit 0 on success.

Kong tags

internal/cmd/claude/config/install.go GO 
type installCmd struct {
    Scope    string   `help:"Settings scope: project or user." default:"project" enum:"project,user"`
    Only     []string `help:"Comma-separated list of items to install (hook event slugs and/or 'status-line')." sep:","`
    Log      bool     `help:"Bake --log into generated command strings."`
    LogLevel string   `help:"Bake --log-level=LEVEL into generated command strings (implies --log)." placeholder:"LEVEL"`
    DryRun   bool     `help:"Print resulting JSON to stdout instead of writing." name:"dry-run"`
}

func (c *installCmd) Run(stdout io.Writer, fs afero.Fs, logger *slog.Logger) error { /* ... */ }

Flow

  1. Resolve the path for --scope (see section 10).
  2. Read the existing file via afero.ReadFile. If it does not exist, start from an empty Settings.
  3. Unmarshal into Settings using jsonv2.Unmarshal. Foreign top-level keys land in the inlined Foreign field as jsontext.Value.
  4. Compute the install set: --only if specified, otherwise {all 25 hook events, status-line}. Validate every token. Reject unknown tokens with an error listing the valid set.
  5. For each token in the install set, build the entry and merge it (see section 08).
  6. Marshal the resulting Settings back to JSON with two-space indent (matches the README's hand-written examples).
  7. If --dry-run, write to stdout and return. Otherwise afero.WriteFile at mode 0644, creating parent directories as needed.
  8. Log a summary at info level: scope, file path, count of entries written, whether logging was baked in.

Examples

examples.sh SHELL 
# Wire everything into the current project's .claude/settings.json
scut claude config install

# Only the formatter + status line, project scope
scut claude config install --only=post-tool-use,status-line

# User scope with debug logging baked in
scut claude config install --scope=user --log-level=debug

# See what would be written without touching the file
scut claude config install --dry-run

// 05uninstall

scut claude config uninstall removes scut entries from settings.json, preserving any non-scut content.

Flags

FlagTypeDefaultEffect
--scope enum {project, user} project Target file (same as install)
--only []string (comma-sep) empty = all scut entries Whitelist of items to remove. Same token set as install --only.
--dry-run bool false Write the resulting JSON to stdout without touching the file

Flow

  1. Resolve the path for --scope. If the file does not exist, write a friendly message to stderr and exit 0 — nothing to remove.
  2. Read and unmarshal as in install.
  3. Compute the remove set (same defaulting + validation as install).
  4. For each hooks[Event] array, drop entries whose inner hooks[].command begins with scut (token boundary aware — see section 08). If a hook array becomes empty after removal, delete the hooks[Event] key entirely.
  5. If statusLine.command starts with scut and status-line is in the remove set, delete the statusLine key.
  6. If the resulting hooks map is empty, delete it too. (Keeps foreign-only files clean.)
  7. If --dry-run, write to stdout. Otherwise afero.WriteFile at mode 0644.
noteuninstall only removes entries whose command begins with scut (followed by space) or is exactly scut. Foreign tools that happen to embed the string scut elsewhere in their command are untouched.

// 07Settings Model

The Go model for settings.json needs three properties: type safety on the keys scut owns, lossless round-tripping of keys scut does not own, and deterministic output for diff-friendly file writes.

encoding/json/v2 with the inline tag option (verified via GOEXPERIMENT=jsonv2 go doc encoding/json/v2: "The inlined field must be a Go struct, jsontext.Value, map[~string]T, or an unnamed pointer to such types. Inlined fields of type jsontext.Value and map[~string]T are called 'inlined fallbacks' as they can represent all possible JSON object members not directly handled by the parent struct.") gives us exactly this:

internal/cmd/claude/config/settings.go GO 
//go:build goexperiment.jsonv2

package config

import (
    "encoding/json/jsontext"
    json "encoding/json/v2"
)

// Settings is the subset of Claude Code's settings.json that scut manipulates.
// Foreign top-level keys round-trip through the Foreign field.
type Settings struct {
    StatusLine *StatusLine             `json:"statusLine,omitzero"`
    Hooks      map[string][]HookGroup `json:"hooks,omitzero"`
    Foreign    map[string]jsontext.Value `json:",inline"`
}

type StatusLine struct {
    Type    string `json:"type"`
    Command string `json:"command"`
}

// HookGroup is one entry in a settings.json hook array (e.g. settings.json#/hooks/PostToolUse[0]).
type HookGroup struct {
    Matcher string     `json:"matcher,omitzero"`
    Hooks   []HookEntry `json:"hooks"`
}

type HookEntry struct {
    Type          string `json:"type"`
    Command       string `json:"command"`
    StatusMessage string `json:"statusMessage,omitzero"`
}
noteThe Hooks field is keyed by Claude Code's event name (e.g. "PostToolUse", "SessionStart") — the same casing the README uses. The map value is an array because Claude Code allows multiple matcher groups per event. Botctrl owns at most one matcher group per event (its own).

// 08Merge Semantics

The defining property: install is idempotent and non-destructive. Running install twice produces the same file. Running install on a file with foreign content preserves that content byte-equivalent in foreign positions.

Ownership rules

Region of settings.jsonscut owns?Behavior on install / uninstall
Top-level keys other than hooks and statusLine no Round-tripped verbatim via Foreign map[string]jsontext.Value
statusLine when its command starts with scut yes install: replaces with scut's entry. uninstall: deletes the statusLine key.
statusLine when its command does not start with scut no install: refuses with an error (do not silently overwrite the user's status line). The user must explicitly remove it or pass --only excluding status-line. uninstall: leaves alone.
hooks[Event][i] where every inner hooks[].command starts with scut yes install: replaces the group. uninstall: removes the group.
hooks[Event][i] where any inner hooks[].command is foreign no Round-tripped verbatim. install appends scut's own group as a new element (rather than touching the foreign one). uninstall ignores it.

The scut prefix test

A command string is "scut-owned" if and only if, after trimming leading whitespace, it equals scut or starts with scut followed by a space, tab, or end-of-string. This rejects commands like not-scut ... and scutsomething while accepting scut claude hook post-tool-use, scut claude --log hook post-tool-use, and /full/path/to/scut status-line if we later choose to support absolute paths (see section 16).

internal/cmd/claude/config/ownership.go GO 
// owns reports whether command is a scut invocation we should manage.
// First scope: exact "scut" or "scut " prefix on a leading token.
func owns(command string) bool {
    c := strings.TrimLeft(command, " \t")
    if c == "scut" {
        return true
    }
    return strings.HasPrefix(c, "scut ") || strings.HasPrefix(c, "scut\t")
}

Output formatting

Determinism invariant: running install twice with the same inputs must produce byte-identical files. This is the property that makes idempotence testable via cmp.Diff on the written bytes (see section 14).

  • Indent: two spaces (matches the README's hand-written examples).
  • Trailing newline: file ends with a single \n (POSIX text file convention).
  • Top-level key order: fixed by the implementation: first statusLine (if present), then hooks (if present), then every key of Foreign in alphabetical order. Go map iteration is randomised, so the implementation must sort foreign keys at marshal time — relying on jsonv2's iteration order would produce non-deterministic output and break the idempotence test.
  • hooks entry order: event-name keys are written alphabetically. Inside each event's array, scut's own group is always first; foreign groups follow in their original order.
  • HookGroup.Hooks order: as written in section 09's registry — one entry per event.

Marshalling is wrapped in a helper marshalSettings(s Settings) ([]byte, error) that performs the sort and applies the indent and trailing newline. Every write path goes through this helper so determinism is enforced at a single point.

Command-string construction

Composed at install time from the install flags:

command-template.txt ASCII 
Hook:        scut claude [--log | --log-level=LEVEL] hook <slug>
Status line: scut claude [--log | --log-level=LEVEL] status-line

Rules:

  • --log-level implies --log: emit --log-level=LEVEL alone (matches internal/cmd/claude/claude.go:25-34; --log would be redundant).
  • Neither flag set: omit both — bare scut claude hook <slug>.
  • --log alone: emit --log.

// 10Scope Resolution

ScopePathResolution
project .claude/settings.json Relative to os.Getwd() at command time. No .git walk-up — Claude Code itself uses .claude/settings.json relative to where it is launched (per README.md:36).
user ~/.claude/settings.json os.UserHomeDir() + .claude/settings.json (per README.md:36 and the standard Claude Code documentation).

Parent directory creation: if the parent (.claude/ or ~/.claude/) does not exist, install creates it with mode 0755 before writing. uninstall and status never create directories.

Resolution is implemented as two single-purpose helpers rather than one function with three same-typed string parameters (which would be indistinguishable at call sites):

internal/cmd/claude/config/scope.go GO 
// projectSettingsPath returns the project-scope settings path relative to cwd.
func projectSettingsPath(cwd string) string {
    return filepath.Join(cwd, ".claude", "settings.json")
}

// userSettingsPath returns the user-scope settings path. It returns an error
// only when os.UserHomeDir fails.
func userSettingsPath() (string, error) {
    home, err := os.UserHomeDir()
    if err != nil {
        return "", fmt.Errorf("resolving user home: %w", err)
    }
    return filepath.Join(home, ".claude", "settings.json"), nil
}

The caller (each leaf command's Run) switches on --scope:

scope-dispatch.go GO 
var path string
switch c.Scope {
case "project":
    cwd, err := os.Getwd()
    if err != nil {
        return fmt.Errorf("resolving cwd: %w", err)
    }
    path = projectSettingsPath(cwd)
case "user":
    path, err = userSettingsPath()
    if err != nil {
        return err
    }
}

Both helpers are pure functions of their inputs (projectSettingsPath takes a string; userSettingsPath reads os.UserHomeDir once). They are trivially unit-testable without mocking.

// 11Error Handling

Sentinel errors

Programmatically-detectable error conditions are exported as sentinels so tests (and any future programmatic consumer) can use errors.Is instead of string matching on err.Error() — error strings are not API.

internal/cmd/claude/config/errors.go GO 
// ErrForeignStatusLine is returned by install when settings.json already
// has a statusLine entry whose command does not start with "scut ".
// Callers must either remove the foreign entry manually or pass --only
// excluding status-line.
var ErrForeignStatusLine = errors.New("settings.json has a non-scut statusLine")

// ErrUnknownOnlyToken is returned when --only contains a token that is
// neither a registered hook slug nor the literal "status-line". The wrapped
// error message lists every valid token alphabetically.
var ErrUnknownOnlyToken = errors.New("unknown --only token")

Wrap sentinels with %w at the use site so the outer message can carry path or token context while errors.Is still walks the chain:

install-wrap.go GO 
return fmt.Errorf("%w at %q: pass --only without status-line to skip, or remove it manually first",
    ErrForeignStatusLine, path)

Condition matrix

All path interpolations use %q so empty paths and paths containing whitespace are visible in the output (per core.md).

Assertion style

Existing tests in this repo (see internal/cmd/claude/hook/posttooluse_test.go) use plain t.Errorf / t.Fatalf with %q for strings and %v for errors. New tests follow the same idiom:

assertion-idiom.go GO 
if got != want {
    t.Errorf("Run() wrote %q, want %q", got, want)
}
if !errors.Is(err, ErrForeignStatusLine) {
    t.Fatalf("Run() error = %v, want ErrForeignStatusLine", err)
}

No cmp.Diff / cmp.Equal: github.com/google/go-cmp is a transitive dependency (via go-git) but not a direct one, and CLAUDE.md requires explicit approval to add direct dependencies. Use t.Errorf with explicit messages instead — they remain self-diagnosing per testing.md.

Error semantics: compare with errors.Is / errors.As, never by matching err.Error(). Sentinels in errors.go make this straightforward.

// 16Out of Scope

Recorded so the implementer does not silently expand scope.

  • Absolute paths in generated commands. The generated command string is always bare scut ..., never /usr/local/bin/scut .... Users who need an absolute path can hand-edit; absolute-path detection in owns() is reserved for a future iteration.
  • Migrating existing user settings. If a user has a hand-written settings.json with non-scut statusLine, install refuses rather than rewriting. Migration tooling is a separate problem.
  • Per-event matcher customization via CLI. The matcher for each event is hard-coded in hookSpecs (e.g. Write|Edit for PostToolUse). Custom matchers require a code change to the registry, not a CLI flag.
  • Watching settings.json for drift. No file watcher; status is a point-in-time read.
  • Restoring from backup. install does not create a settings.json.bak. Users versioning .claude/settings.json in git get the same benefit for free.
  • Validating that scut is on PATH. The generated command assumes scut is discoverable when Claude Code spawns the subprocess — same assumption the existing README documentation makes.
